# Findings: 2026-05-19 RowBlock2D Scatterv + Post-Use Reduction

## 2026-05-27 Four-Case Shenwei Efficiency Diagnosis
- Fresh large-case `--mode=buffer --mpi` outputs were generated under `/Volumes/QUQ/working/analysis_tmp/four_case_codegen` for `imageAdjustment1.0`, `waveEquation1.0`, `stencil1.0`, and `FOuLa1.0`.
- Common buffer-mode issue: generated kernels recreate `sycl::buffer` objects around host `std::vector` storage for every kernel submission, then call `q.wait()`. MPI halo and boundary replay happen on host vectors. On an accelerator backend this implies frequent host/device synchronization and data migration instead of persistent device-resident arrays.
- Common temporal issue: stencil-like translations use `temporal-block=2`, which halves halo frequency but still launches one SYCL kernel per original time step. `waveEquation1.0` and `stencil1.0` at 500 steps still submit about 500 kernels; `FOuLa1.0` at 100000 steps still submits about 100000 tiny kernels.
- `imageAdjustment1.0`: translation fuses the two pointwise RowBlock2D kernels and constant-fills the input locally, so the useful work is only one very small 64x64 kernel. It still performs a final `MPI_Gatherv` of the full `Pixel` image and root `array2Tensor` because `image_tensor3.print()` is a full tensor post-use. Multi-node efficiency is therefore dominated by fixed launch/MPI/root overhead rather than compute.
- `waveEquation1.0`: current translation accepts `distribution=spatial-2d` with `temporal-block=2` and direct-reader role rotation. Init skips `matCur` scatter due constant zero but still root-packs/scatters `matPrev`. Each 2-step block performs one spatial halo exchange plus two kernel submissions. Spatial halo runtime dynamically packs/unpacks edges and corners into temporary vectors and posts up to 8 receives plus 8 sends per rank/block. Boundary replay is emitted as serial host loops over full global boundary ranges before/after kernels. Final materialization gathers owned final state to root, patches a full root copy of `matCur`, replays boundaries again, and calls `array2Tensor` because `matCur.print()` is classified as full post-use.
- `stencil1.0`: current translation also accepts `spatial-2d` with `temporal-block=2`. It root `tensor2Array`s and spatial-scatters the initial 2048x2048 input, then for each 2-step block performs one spatial halo exchange, two kernel submissions, and host boundary replay loops. Final materialization gathers owned output to root and patches a full `matIn` copy because `matIn[0].print()` is not currently accepted as root-observable bounded output.
- `FOuLa1.0`: owner-loop specialization is structurally good for final sync: only fixed row 1 history is kept and sent to root if needed. However, the generated owner loop redundantly calls both `scatter_window_1d` and `scatter_window_1d_temporal` on the same initial column before time stepping. With `m=2000` and many nodes, each rank owns only tens of spatial points; with `n=100000` and `temporal-block=2`, the code still performs about 50000 halo exchanges and 100000 tiny kernel submissions, so launch and MPI latency dominate.
- Runtime-specific issue for `spatial-2d`: `exchange_halo_2d_spatial_inplace` uses runtime allocation/packing for north/south/east/west and four corner messages. This reduces bytes compared with full-row halos but increases message count and host pack/unpack overhead, which is a poor tradeoff on latency-sensitive multi-node runs.
- Final-sync issue for `spatial-2d`: `gather_spatial_owned_to_root` gathers rank-local rectangles into a rank-contiguous buffer and then root repacks into a dense global matrix. For full post-use cases this adds root-only O(NX*NY) work and serial memory traffic after the distributed computation.

## 2026-05-22 stencil1 spatial-k2 profitability follow-up findings
- Current HEAD rejects `stencil1.0` spatial-k2 because `matIn` has host post-use and `annotateResidentHaloSpatial2D` requires `postUseSync=None` for `temporalBlockSize > 1`.
- The row-temporal fallback already avoids a full gather for the large profile form: `matIn post-use=bounded-index count=1`, `matOut post-use=none`, no gather segment, and materialize is microsecond-scale. The remaining cost is halo/kernel, not final full materialization.

This is not an acceptable formal gate for `stencil1.0` tests unless full post-use remains rejected.
- For the large profile path, the script rewrites the post-use to a bounded root read, so the experimental spatial-k2 profile isolates halo/layout cost rather than full gather cost.
- Experimental bounded spatial-k2 profile in `/Volumes/QUQ/working/mpi_tmp/profile_segments_p11_stencil1_spatial_k2_experiment` was correct but not profitable: `dac_wall_median_s=4.838048`, profile median `4.791256`, kernel median max `4496.731972 ms`, materialize median max `0.000666 ms`. This proves final-sync/post-use is not the bottleneck; the current spatial rectangular k=2 kernel/layout path is.
- Formal path should therefore retain row-temporal k=2 for any k=2 host post-use until the spatial kernel/layout cost is fixed. No-use contracts such as `mpiLoopStencilCountGuard2D` can continue accepting spatial-k2 because they avoid this host-visible final-state profitability case.

## 2026-05-22 P11 2D Spatial Block Partition Findings
- Current `StencilWindow2D` resident halo uses `ResidentHalo2DRowLayout`: contiguous row-slab ownership, full-width row halos, and k=2 temporal blocking for canonical B2/B3 shapes.
- `stencil1.0` is canonical B2: one 3x3 stride-1 window reader, one output-direct writer over the interior, writer->reader followup offset `(1,1)`, and four boundary-local self-copy loops. It has no direct reader and is the safest target for a first spatial-2d resident-halo mode.
- `waveEquation1.0` is B3 direct-reader recurrence and should remain row fallback until 2D role rotation/read-cache semantics are proven on a rectangular process grid.
- For spatial k=2, `resident_halo_2d_spatial_layout` and `scatter_window_2d_spatial` already accept a halo-width parameter, but `exchange_halo_2d_spatial_inplace` is currently hardcoded to one row/column plus 1x1 corners. The k=2 implementation must widen edge and corner transfers to width 2.
- The codegen can reuse the existing local halo-buffer indexing for B2 spatial k=2: after one width-2 exchange, step 0 computes an expanded rectangle one cell beyond the owned output rectangle into the writer/intermediate buffer, boundary replay refreshes the intermediate, then step 1 computes only the owned rectangle back into the reader/latest buffer with no MPI inside the block.
- Existing temporal blocking assumes row-only widened halos. P11 first cut should disable k=2 when spatial-2d is accepted and report a single-step spatial block partition; row fallback keeps temporal-blocked paths intact.
- Runtime needs a rectangle layout with rank-grid dims, owned row/col ranges, local reader rectangle including one-cell halo on all sides, neighbor ranks, rectangle scatter, owned-rectangle slice for gathers, and bounded root-read owner/local-slot mapping.

## RowBlock2D Input Path
- `CollectiveCodegenUtils.cpp::emitScatter` currently has a RowBlock2D byte fast path named `canUseRowBlockByteBroadcast`.
- That fast path checks 2D contiguous row-major tensor layout on root, then broadcasts the entire dense tensor and copies each rank's row block locally.
- Non-byte RowBlock2D currently also broadcasts the full dense tensor and copies the local row block.
- `PartitionCodegen.cpp` derives `__or_rows`, `__or_cols`, `__or_total_items`, `__or_row_counts`, `__or_row_displs`, and flattened `__or_counts/__or_displs`.
- `byte_counts_displs` already uses checked multiply and narrow helpers; RowBlock2D element counts currently use unchecked `row_count * cols` casts and should be tightened while adding direct Scatterv.
- Final implementation emits a guarded direct `MPI_Scatterv` for RowBlock2D row-major contiguous tensors. The guard requires 2D shape match, nonnegative offset, row-major strides, storage capacity for `offset + __or_total_items`, and valid per-rank row ranges.
- The byte path scatters with `MPI_BYTE` and checked byte counts/displacements. The non-byte path scatters with the typed MPI datatype. If the guard fails, byte transport falls back to pack + Scatterv and non-byte transport falls back to the previous full Bcast + local copy behavior.
- Target and full profiles show imageAdjustment's first RowBlock2D wrapper has a scatter segment and no input bcast segment.

## WaveEquation Communication Baseline
- `waveEquation1.0` currently lowers to OR 2D resident halo B3. The generated code initializes once, runs 600 `_run` iterations, and materializes once at the end.
- For the benchmark scale `NX=NY=2048`, `TIME_STEPS=600`, 4 ranks, double elements:
  - Initial distribution sends the window reader `matCur` as row windows, roughly 24.1 MiB over the network to non-root ranks, plus direct-reader `matPrev` owned interior rows, roughly 23.9 MiB.
  - Each iteration exchanges top/bottom row halos with neighboring row-block ranks. Network payload is `(ranks-1) * 2 * (NY-2) * sizeof(double)` = 98,208 bytes per step for 4 ranks, about 56.2 MiB across 600 steps.
  - Final materialization gathers two interior tensors (`matNext/current` and `matPrev`), roughly 47.9 MiB over the network to root for 4 ranks.
  - Rough network payload at 4 ranks is about 152 MiB, excluding MPI protocol overhead and local root copies.
- Full profile for `waveEquation1.0` after current changes: wall median 1.134183 s, profile median 1.079786 s; segments show halo 2400 calls with median max 146.771 ms and final gather 8 calls with median max 17.919 ms.
- Main communication optimization opportunities are: avoid final gathers when the only host-visible post-use is a small bounded read/print, avoid initial root packing/scatter for zero/known tensors, and reduce halo message overhead for high-rank runs by using persistent/nonblocking neighbor exchange or a distributed graph/cartesian communicator.

## 2026-05-21 Top3 Optimization Discovery
- `matMul1.0` currently lowers to `RowPartitionFullRow` OR and still emits root `tensor2Array` plus `MPI_Scatterv` for output `dotProduct`/`matC`. The plan marks `matC` as `reads=1 writes=1` because the calc uses `dotProduct[0] += ...`, but non-MPI buffer lowering already uses `discard_write` and leaves the output buffer uninitialized; benchmark semantics therefore match default/local output initialization rather than preserving old `matC` values.

The current DFT miss is twofold: the accepted layout gate only allows `Contiguous1D`, while DFT's shell layout is `ReplicatedFullTensor`; and helper-function wrappers fail because `analyzeConstantInitForShellArg` only scans `main`.

There is no existing fused wrapper; generated code has two `parallel_for` submissions and the intermediate full local buffer is resident.

## 2026-05-22 Mandel Contiguous1D Block-Cyclic Findings

- Current mandel generated code uses `rank_range_1d`, `MPI_Scatterv` for `complex_points`, local default output, one SYCL kernel over `__or_local_item_count`, and `MPI_Reduce` for `mandelbrot_count`.
